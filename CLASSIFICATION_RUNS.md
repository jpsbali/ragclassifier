# Classification Run Logging Architecture

## Objective
To implement a robust logging system that generates a single `results_<date>.xlsx` file per day. This file will aggregate all classification runs, enabling detailed comparison of different Large Language Models (LLMs), cost analysis, and performance tracking.

## Phase 1: Data Schema Design
To satisfy the requirement of comparing different LLMs and tracking costs, the schema includes specific metadata columns alongside standard classification results.

### 1. Run Metadata (Crucial for LLM Comparison)
*   **`timestamp`**: Exact date and time of the run (ISO 8601 format recommended).
*   **`model_name`**: The specific LLM used (e.g., `gemini-1.5-pro`, `gpt-4o`, `claude-3-opus`).
*   **`supervisor_temp`**: Temperature setting for the supervisor model.
*   **`agent_a_temp`**: Temperature setting for agent A.
*   **`agent_b_temp`**: Temperature setting for agent B.
*   **`prompt_version`**: Identifier for the prompt template used (enables A/B testing of prompts).

### 2. Input Data
*   **`document_id`**: Unique identifier for the document.
*   **`document_name`**: Original filename of the document.
*   **`file_names`**: List of specific files processed in this run (if multiple files constitute one document).

### 3. Classification Results
*   **`classification`**: The resulting category or label assigned by the LLM.
*   **`confidence`**: Confidence score (normalized to 0.0 - 1.0).
*   **`consensus_reached`**: Boolean (`True`/`False`) indicating if a consensus was reached (if using multi-agent/voting).
*   **`rounds_used`**: Integer indicating how many iterations/rounds the LLM took to reach a conclusion.
*   **`risk_flag`**: Risk level indicator (e.g., `High`, `Medium`, `Low`, or Boolean).

### 4. Cost & Metrics
*   **`input_tokens`**: Count of tokens sent to the model.
*   **`output_tokens`**: Count of tokens generated by the model.
*   **`estimated_cost`**: Calculated cost based on token counts and specific model pricing.
*   **`risk_cost`**: Potential financial or operational cost associated with the risk level.

---

## Phase 2: File Management Strategy
Since Excel files (`.xlsx`) are binary, the system cannot simply append text to the end of the file. The workflow requires a Read-Update-Write cycle.

### Workflow Logic
1.  **Path Generation**: Dynamically generate the filename based on the current date:
    *   Format: `results/results_YYYY-MM-DD.xlsx`
2.  **Existence Check**:
    *   **If File is Missing**: Initialize a new DataFrame/Workbook with the headers defined in Phase 1.
    *   **If File Exists**: Load the existing Excel file into memory.
3.  **Data Appending**: Append the new row(s) of run details to the in-memory dataset.
4.  **Write Back**: Save the updated dataset back to the `results` folder, overwriting the previous version of that day's file.

---

## Phase 3: Concurrency & Safety
To prevent data corruption when multiple classification processes run simultaneously (e.g., multi-threading or parallel scripts), a file locking mechanism is required.

### Mechanism
1.  **Lock File**: Use a temporary lock file (e.g., `results_2023-10-27.xlsx.lock`).
2.  **Acquisition Logic**:
    *   Before reading the Excel file, check for the existence of the lock file.
    *   **If Locked**: Wait/Sleep for a random short interval (backoff strategy).
    *   **If Unlocked**: Create the lock file immediately.
3.  **Release Logic**:
    *   Perform the Read -> Append -> Write operation.
    *   Delete the lock file immediately after the write operation completes.

---

## Phase 4: Analysis & Comparison Strategy
The data structure is designed to facilitate immediate Pivot Table analysis in Excel or programmatic analysis in Python.

### Key Comparison Metrics
*   **Cost vs. Accuracy**: Compare `estimated_cost` against `confidence` (or `ground_truth` if available) grouped by `model_name`.
*   **Efficiency**: Compare `rounds_used` and `execution_time` across different `model_name` and `prompt_version` combinations.
*   **Stability**: Analyze `consensus_reached` rates to determine which models are more consistent.

### Future-Proofing
*   **Ground Truth Column**: Include an empty `ground_truth` column by default. This allows human reviewers to manually enter the correct classification later, enabling automated accuracy percentage calculations.
